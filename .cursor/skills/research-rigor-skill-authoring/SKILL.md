# 科研严谨与 Skill 编写规范

## 适用范围

在以下场景优先应用本技能：
- 用户提到论文、实验、方法、结果、讨论、统计显著性、复现、可靠性。
- 用户要求创建或修改 Cursor Skill（例如 `create-skill`、`SKILL.md`、技能目录结构）。
- 用户要求“严格执行规范”“不能臆测”“按科研标准输出”。

## 核心原则（必须遵守）

1. **禁止臆测**：没有证据就明确写“证据不足/待验证”，不得补全未知事实。
2. **证据优先**：结论必须绑定可追溯证据（数据文件、实验记录、代码或文献来源）。
3. **可复现**：涉及实验时给出最小复现信息（数据版本、参数、环境、随机种子、评估指标）。
4. **统计合规**：对“提升/差异显著”类表述，需给出检验方法、样本量、置信区间或 p 值。
5. **局限透明**：必须列出边界条件、潜在偏差、失败案例和外推风险。
6. **中文输出**：默认中文；技术名词可保留英文括注。
7. **编码一致**：涉及脚本或命令时，默认 UTF-8。

## 工作流程 A：论文与科研内容输出

按以下顺序执行，不得跳步：

1. **需求澄清**
   - 识别任务类型：写作、改写、审稿、补实验设计、结果解释。
   - 明确输入边界：仅基于已给材料，是否允许使用外部文献。

2. **证据盘点**
   - 列出可用证据清单：数据、图表、日志、代码、文献。
   - 标记缺口：哪些结论缺少直接支持。

3. **结构化产出**
   - 强制使用模板，见 [TEMPLATE.md](TEMPLATE.md)。
   - 每条关键结论附“证据锚点”（文件名/表格名/实验编号/文献链接）。

4. **严谨性自检**
   - 执行 [CHECKLIST.md](CHECKLIST.md) 中“科研输出检查清单”。
   - 任一关键项不满足时，不输出“确定性结论”，改为“待验证结论”。

## 工作流程 B：创建或更新 Cursor Skill

当用户要求创建 Skill 时，按此流程：

1. **需求采集（最少 5 项）**
   - 目标任务与边界
   - 存储位置（个人级或项目级）
   - 自动触发场景关键词
   - 输出格式与质量标准
   - 是否需要脚本/参考文档

2. **技能设计**
   - `name` 使用小写+连字符，长度不超过 64。
   - `description` 必须包含 WHAT + WHEN（做什么 + 何时触发）。
   - 主文件保持精简，详细内容放到一层引用文件。

3. **目录实现**
   - 标准结构：
     - `skill-name/SKILL.md`
     - 可选：`skill-name/reference.md`
     - 可选：`skill-name/examples.md`
     - 可选：`skill-name/scripts/...`

4. **质量验证**
   - `SKILL.md` 不超过 500 行。
   - 术语一致，避免含糊命名。
   - 不写过时的时间敏感规则。
   - 文件引用保持一层深度。

## 输出约束（强模板 + 引证规则）

- 默认使用 [TEMPLATE.md](TEMPLATE.md) 的“科研严谨输出模板”。
- 每个一级结论至少绑定 1 个证据锚点。
- 外部信息必须给出可访问链接；无法核验时显式标注“未核验”。
- 不把猜测写成事实，不把相关性写成因果性。

## 失败与升级策略

遇到以下情况必须先提示再继续：
- 输入材料不足以支持用户要求的确定性结论。
- 实验描述缺少关键复现参数（数据版本、随机种子、评估口径）。
- 统计结论缺少显著性检验信息。

提示格式：
- `风险`：当前结论可能不可靠的原因。
- `最小补充项`：需要用户提供的最少信息。
- `可先执行项`：在现有材料下仍可完成的安全输出范围。

## 术语约定

- 统一使用“证据锚点”（不要混用“依据”“支撑点”“证明材料”）。
- 统一使用“可复现信息”（不要混用“环境描述”“运行细节”）。
- 统一使用“待验证结论”（不要混用“猜测”“大概结论”）。
